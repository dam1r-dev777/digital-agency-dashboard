# Payment Dashboard

Дашборд для контроля документооборота digital-агентства: связывание входящих платежей с проектами и мониторинг статусов закрывающих документов (актов).

## Бизнес-задача

### Проблема

Digital-агентство работает с десятками клиентов одновременно. Оплата за проекты поступает регулярно, но закрывающие документы — акты выполненных работ — нередко задерживаются: менеджер забыл отправить, клиент тянет с подписью, или акт потерялся в переписке.

Это порождает **кассовые разрывы**: деньги на счету есть, а в бухгалтерском учёте выручка не закрыта. При налоговой проверке или аудите это критично. Кроме того, клиент, не получивший акт вовремя, может оспорить факт оказания услуги.

### Решение

Дашборд даёт операционному менеджеру единое окно для контроля:

- каждый поступивший платёж автоматически сопоставляется с проектом и клиентом;
- для каждого платежа видно, на каком этапе находится акт (`Не отправлен → Ожидает подписи → Закрыт`);
- система **автоматически** помечает проблемные акты статусом **«Требует внимания»**, если:
  - прошло более **14 дней** с момента платежа, а акт так и не отправлен,
  - или акт отправлен, но не подписан более **30 дней**;
- менеджер меняет статус акта в один клик прямо из таблицы, не переключаясь между системами;
- к каждому акту можно оставить текстовый комментарий для внутренних заметок.

## Технический стек

| Слой | Технологии |
|------|------------|
| **Backend** | Python 3.11, FastAPI 0.111, SQLAlchemy 2.0 (async), Alembic, Pydantic v2, asyncpg |
| **Frontend** | Next.js 14 (App Router), React 18, TypeScript, TailwindCSS, SWR |
| **База данных** | PostgreSQL 15 |
| **Инфраструктура** | Docker, Docker Compose |

### Ключевые архитектурные решения

**Статус акта вычисляется в Python, а не хранится в БД.** Статус `REQUIRES_ATTENTION` зависит от `datetime.now()`, который меняется каждую секунду. Хранить его в базе потребовало бы фоновых джобов или триггеров. Вместо этого он вычисляется в Pydantic `model_validator` при каждом формировании ответа — без дополнительной нагрузки на БД.

**Proxy-rewrite в Next.js.** Браузер обращается только к одному хосту (`localhost:3000`), Next.js сервер проксирует `/api/*` на бэкенд внутри Docker-сети (`http://backend:8000`). Никакого CORS в продакшн-сценарии.

**Optimistic UI в таблице.** `ActStatusBadge` немедленно обновляет локальный стейт при клике, отправляет `PATCH`-запрос в фоне и откатывает изменение при ошибке. Таблица не мигает и не блокируется.

**Два драйвера для одной БД.** SQLAlchemy async-движок использует `asyncpg` в рантайме. Alembic не поддерживает async-драйверы, поэтому для него прописан отдельный `SYNC_DATABASE_URL` с `psycopg2`. Обе переменные передаются через `docker-compose.yml`.

## Сущности и связи

```
clients
  │
  ├──< projects (client_id → clients.id)
  │        │
  │        └──< payments (project_id → projects.id)
  │
  └──< payments (client_id → clients.id)
           │
           └──1 acts (payment_id → payments.id, UNIQUE)
```

Каждый платёж принадлежит конкретному клиенту и проекту. У каждого платежа может быть ровно один акт (отношение 1:1, `UNIQUE` constraint на `acts.payment_id`). Если акт ещё не создан, платёж считается в статусе `NOT_SENT`.

### Таблица `clients`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID PK | |
| `name` | VARCHAR(255) | Название компании / ИП |
| `inn` | VARCHAR(12) UNIQUE | ИНН |
| `ogrn` | VARCHAR(15) | ОГРН |
| `bank_account` | VARCHAR(25) | Расчётный счёт |
| `contact_person` | VARCHAR(255) | Контактное лицо |

### Таблица `projects`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID PK | |
| `client_id` | UUID FK → clients | |
| `name` | VARCHAR(255) | Название проекта |
| `status` | ENUM | `active` / `paused` / `completed` |

### Таблица `payments`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID PK | |
| `project_id` | UUID FK → projects | |
| `client_id` | UUID FK → clients | |
| `payment_date` | DATE | Дата зачисления |
| `amount` | NUMERIC(15,2) | Сумма в рублях |
| `payment_purpose` | VARCHAR(500) | Назначение платежа |
| `service_stage` | VARCHAR(255) | Этап работ |

### Таблица `acts`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID PK | |

| `is_signed` | BOOL | Акт подписан клиентом |
| `signed_at` | TIMESTAMPTZ | Проставляется автоматически |
| `manager_comment` | TEXT | Произвольный комментарий менеджера |

## Функциональность

### Автоматический расчёт статуса акта

Статус вычисляется в Pydantic `model_validator` (файл `backend/app/schemas/act.py`) при каждом запросе. В БД статус **не хранится**.

```
┌─────────────────────────────────────────────────────────────────┐
│  REQUIRES_ATTENTION  — прошло > 14 дней с payment_date,        │
│                        а акт не отправлен                       │
│                      — ИЛИ акт отправлен, но не подписан       │
│                        более 30 дней                            │
├─────────────────────────────────────────────────────────────────┤
│  CLOSED              — is_sent = true  И  is_signed = true      │
├─────────────────────────────────────────────────────────────────┤
│  AWAITING_SIGNATURE  — is_sent = true,  is_signed = false       │
│                        (и не попадает в REQUIRES_ATTENTION)     │
├─────────────────────────────────────────────────────────────────┤
│  NOT_SENT            — is_sent = false                          │
│                        (или запись в acts отсутствует)          │
└─────────────────────────────────────────────────────────────────┘
```

`payment_date` доступен в схеме акта через атрибут `act._payment_date`, который роутер проставляет перед валидацией (`act._payment_date = payment.payment_date`).

### Быстрые действия в таблице

Прямо в колонке «Статус акта» менеджер видит кнопки-ссылки для перехода на следующий этап:

| Текущий статус | Доступное действие |
|----------------|--------------------|
| `NOT_SENT` (акта нет) | → Отправить акт *(создаёт запись акта с `is_sent=true`)* |
| `NOT_SENT` (акт есть, `is_sent=false`) | → Отправить акт |
| `REQUIRES_ATTENTION` с `is_sent=false` | → Отправить акт |
| `AWAITING_SIGNATURE` | ✓ Подписать акт |
| `REQUIRES_ATTENTION` с `is_sent=true` | ✓ Подписать акт |

Клик выполняет оптимистичное обновление локального стейта, затем отправляет `PATCH`-запрос на бэкенд. При ошибке стейт откатывается с показом сообщения.

### Поиск по назначению платежа и клиенту

В панели фильтров (`FilterPanel`) есть текстовое поле «Поиск». Запрос уходит на бэкенд параметром `search`, где выполняется `JOIN` с таблицей `clients` и фильтрация через `ILIKE`:

```sql
WHERE payment_purpose ILIKE '%<query>%'
   OR clients.name    ILIKE '%<query>%'
```

Поиск чувствителен к регистру только на уровне настроек collation PostgreSQL (по умолчанию — нечувствителен).

### Комментарии к актам

Рядом с бейджем статуса отображается иконка карандаша:
- **серая** — комментарий отсутствует, клик открывает пустое поле ввода;
- **синяя** — комментарий уже есть, tooltip показывает его текст.

Клик разворачивает встроенный редактор с `textarea` и кнопками «Сохранить» / «Отмена». Сохранение вызывает существующий `PATCH /api/v1/acts/{id}/status` с полем `manager_comment`. После успешного ответа редактор схлопывается, иконка становится синей.

### Виджеты сводной статистики

Четыре карточки в верхней части дашборда обновляются каждые 30 секунд через SWR:
- Всего платежей / общая сумма
- Количество актов по каждому статусу
- Счётчик «Требует внимания» с акцентным выделением

### Фильтрация и пагинация

| Фильтр | Применяется |
|--------|-------------|
| Дата от / Дата до | SQL `WHERE payment_date BETWEEN ...` |
| Проект | SQL `WHERE project_id = ...` |
| Статус акта | Python-цикл после выборки страницы |
| Поиск (текст) | SQL `JOIN clients WHERE ... ILIKE` |

При любом изменении фильтров пагинация сбрасывается на первую страницу (обработано через `useEffect` в `PaymentsTable`).

| Метод | URL | Описание |

## Запуск для ревьюера

> **Требования:** Docker Desktop ≥ 24, Docker Compose v2 (входит в Docker Desktop).

### Шаг 1 — Создать файл переменных окружения

Создайте файл `.env` в корне репозитория (рядом с `docker-compose.yml`):

```env
POSTGRES_USER=payment_user
POSTGRES_PASSWORD=payment_pass
POSTGRES_DB=payment_db
```

Значения можно оставить как есть — они нужны только для локального контейнера.

---

### Шаг 2 — Собрать и запустить все сервисы

```bash
docker compose up --build
```

Docker Compose поднимет три контейнера в правильном порядке:

1. **`db`** — PostgreSQL 15; health-check ждёт готовности принимать соединения.
2. **`backend`** — FastAPI; при старте автоматически применяет миграции (`alembic upgrade head`), затем запускает Uvicorn.
3. **`frontend`** — Next.js; проксирует `/api/*` на бэкенд внутри Docker-сети.

Дождитесь строки в логах:

```
backend  | INFO:     Application startup complete.
frontend | ✓ Ready in ...ms
```

> Флаг `-d` запускает контейнеры в фоне: `docker compose up --build -d`.  
> Логи тогда смотреть через `docker compose logs -f`.

---

### Шаг 3 — Заполнить базу тестовыми данными и открыть дашборд

```bash
docker compose exec backend python -m app.seed
```

Скрипт создаёт ~4 клиента, ~10 проектов и ~50 платежей с актами в разных статусах (включая просроченные). Запускается однократно; повторный вызов добавит дублирующие записи.

Откройте браузер:

| URL | Что там |
|-----|---------|
| **http://localhost:3000** | Дашборд |
| http://localhost:8001/docs | Swagger UI — ручной вызов API |
| http://localhost:8001/health | JSON-healthcheck бэкенда |
